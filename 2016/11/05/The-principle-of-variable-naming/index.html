The principle of variable naming

Posted on 2016-11-05 Edited on 2019-11-05 In programming

变量命名的原则

类型和变量的命名绝对是平时开发中保证代码整洁、可读性和可维护性的重要环节，每次在新系统的开发中，针对变量、函数和类的命名我都要反复斟酌，以获得最高程度的合理性，下面介绍一些《代码整洁之道》中的变量命名principle

名副其实

名字一定要符合变量的实际含义，体现变量使用的目的。尽量不要使用单字母变量

避免误导

  • 不应该使用现实世界中已有的专有名称，如hp,aix,sco
  • 不应该使用两组差异性太小的单词，很容易造成阅读及使用混乱
  • 变量名中不应该包含类型信息

做有意义的区分

  • 不要使用字母加数字的命名a1,a2,a3…，而应该根据其意义命名
  • 命名中不用加入废话，比如Product好于ProductData/ProductInfo，不需要添加a,an,the等定冠词

不要创造单词


变量需要易于搜索

单字母变量很难在大型工程代码中搜索到想要的代码块，因此除非极短方法中的局部变量，其余都不应该使用单字母进行变量命名。变量必须很容易在工程目录中搜索区分出来。

避免使用编码

  • 不要在名称中加入类型信息，比如phone_number，不要叫phone_number_str
  • 成员变量不需要加m_前缀，可以使用_后缀

隐性约定

  • 类名应该是名词或名词短语， 如Customer,WikiPage,Account,AddressParser
  • 方法名应该是动词或动词短语，如postPayment, deletePage, save, get, set

概念的一致性

同一个概念需要在代码中保持同一个单词，即概念一致性。比如取数据的操作，要么都叫fetch，要么都叫get。同样的，一堆代码中不能既有controller,又有manager还有driver。DeviceManager和ProtocolController之间没有本质区别。

不用双关语

还是一致性的问题，同一个单词不可以在一个代码项目中体现两个不同的概念，其体现的意义必须一致。

使用解决方案领域名称

尽量使用工程所涉及的领域的专用名词来描述问题。或者可以尽量使用计算机科学领域的专用名词

其他

  • 尽量不要添加没有用的语境/前缀
  • 单词的描述尽量保证精确性
  • 不要害怕重命名，想到更好的词要立即替换过来
